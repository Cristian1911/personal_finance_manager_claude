**Per D-05, D-06:** Wrap all recharts-importing components in `next/dynamic` with `ssr: false` at the point where they are imported by parent components. Use the established pattern from `scenario-planner.tsx`.

### Critical rules:
- `ssr: false` is mandatory on every dynamic import (recharts uses DOM APIs)
- Dynamic imports declared at module scope, NOT inside component functions
- Named exports use `.then((m) => ({ default: m.ExportName }))` pattern
- `loading` prop must have a content-shaped skeleton matching the chart height (D-06, D-13)
- For Server Components that import charts: they need `import dynamic from "next/dynamic"` added

However, if `BurnRateSection` is an async Server Component (which it should be from Plan 01), it cannot use `next/dynamic` directly. Instead, create a thin client wrapper or use the dynamic import in the page-level scope and pass it down. The simpler approach: keep the static import in the async sub-component (it runs server-side anyway). The `ssr: false` pattern only makes sense in client components.

**Resolution for Server Component chart imports:** For `presupuesto-section.tsx`, `patrimonio-section.tsx`, and `flujo-charts.tsx` — these are async Server Components. They render chart components that are `"use client"`. In Next.js App Router, when a Server Component imports a Client Component, Next.js automatically code-splits the client component into its own chunk. However, `next/dynamic` with `ssr: false` ensures the chart JS is NOT evaluated during SSR (which recharts needs since it uses DOM APIs). So `next/dynamic` IS needed even in Server Components to prevent SSR evaluation of recharts.
