# Phase 4 — UI Design Contract: Dashboard Performance

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope Note

Phase 4 is a performance refactor — no new screens, no new user-facing features. The only visible output is:

1. **Skeleton components** that stream in before tier 2 data arrives (PERF-01, PERF-05)
2. **CSS step transitions** on the onboarding page replacing framer-motion (PERF-03)
3. **Chart lazy-loading** via `next/dynamic` — invisible to the user except via the skeleton loading prop (PERF-02)
4. **Recharts v3 upgrade** — no visible change if migration is correct (PERF-04)

The design contract therefore focuses primarily on skeleton geometry and the onboarding transition behavior. All other design decisions inherit from the existing Zeta design system in `globals.css`.

## Skeleton Geometry

> This is the primary design output for Phase 4. Each tier 2 dashboard section requires a skeleton that matches the real component's height and layout structure.

### Skeleton Component

Use the existing `<Skeleton>` from `@/components/ui/skeleton`. It applies `bg-accent animate-pulse rounded-md`.

**Override for skeletons:** shadcn's `Skeleton` uses `bg-accent` which maps to `--z-surface-2`. This is correct. Use `<Skeleton className="..." />` or inline `<div className="rounded-md bg-muted animate-pulse" />` — both are acceptable.

### Per-Section Skeleton Specifications

Each skeleton must:
- Match the real component height (prevents CLS — PERF-05, D-13)
- Mirror the structural layout (card header + content area — D-12)
- Use only `animate-pulse` — no additional animation libraries (D-14)

### Shared Skeleton Component Decision

Create a single `webapp/src/components/ui/skeleton.tsx` export (already exists). Implement per-section skeletons as **named exports in a single file**: `webapp/src/components/dashboard/dashboard-skeletons.tsx`. This avoids proliferating `*-skeleton.tsx` files while keeping each skeleton co-located with the dashboard domain.

## Onboarding Step Transition Contract

Replaces `framer-motion` (`motion` + `AnimatePresence`) in `webapp/src/app/onboarding/page.tsx`.

### Behavior

| Trigger | Animation | Implementation |
|---------|-----------|----------------|
| Step mounts | Fade in + slide from right | `animate-in fade-in slide-in-from-right-4 duration-200` |
| Step unmounts | Immediate (no exit animation) | DOM removed; new step's enter animation provides continuity |

### CSS Classes (tw-animate-css — already installed)

```
animate-in fade-in slide-in-from-right-4 duration-200
```

This is a deliberate UX simplification from CONTEXT.md D-08: "Simple fade+slide. Zero JS bundle cost." Exit animations are dropped because they require JS orchestration (`AnimatePresence`) — the simplified approach is acceptable for this app.

### Duration

200ms — fast enough to feel snappy on mobile viewport, slow enough to orient the user between steps.

## Interaction Contract

### Dashboard Loading Sequence

| Tier | Components | Behavior | User Experience |
|------|-----------|----------|-----------------|
| Tier 1 | `DashboardHero`, `HealthMetersCard` | Awaited at page level — renders before streaming begins | Instant paint on navigation |
| Tier 2 | All other sections (burn rate, accounts, cashflow, presupuesto, patrimonio, heatmap, upcoming) | Async Server Components in `<Suspense>` boundaries | Skeleton shown, then replaced when data arrives |

**Per-section Suspense boundaries:** Each dashboard section gets its own `<Suspense>` wrapper (not one for all tier 2 content). This matches D-02: "grouped per dashboard section" to reduce visual churn.

### Chart Loading via next/dynamic

| Trigger | Behavior |
|---------|----------|
| Parent component renders | `next/dynamic` loading prop skeleton shown immediately |
| Chart JS bundle loaded | Skeleton replaced with live chart — no layout shift |

Loading prop skeletons follow the Skeleton Geometry specifications above for each chart.
